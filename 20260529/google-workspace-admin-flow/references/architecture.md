# 系統架構與資料流

## 一、設計理念

**痛點**：學校承辦人收到公文後，資料散落在信件、桌面、雲端、通訊軟體；結案時需要的照片、簽到、回饋、經費憑證臨時找不到。

**解法**：把「公文 → 各種附屬資料」的關係，從**人腦記憶**改成**Drive 結構**。每張公文進來就建一個專案資料夾，所有相關資料都歸位。

**邊界**：系統只做建檔、追蹤、提醒。**不取代人工**：公告、簽核、經費核銷、成果送出仍由承辦人與主管把關。

## 二、資料流

```
[公文 / 任務] ─→ [公文專案啟動表] ─→ onIntakeSubmit
                                          ├─→ Drive 建專案資料夾（11 子資料夾）
                                          ├─→ Docs 建專案紀錄（9 段結構）
                                          ├─→ Sheets 建待辦追蹤表（12 欄）
                                          ├─→ Sheets 建成果檢核表（6 欄）
                                          ├─→ Calendar 建初始提醒
                                          ├─→ 寫入行政專案總控表
                                          └─→ Gmail 通知承辦人

[後續日期] ──→ [專案階段日期新增表] ─→ onMilestoneSubmit
                                          ├─→ 寫入專案階段日期紀錄
                                          ├─→ 追加到該案的待辦追蹤表
                                          ├─→ Calendar 建多個提醒（依設定）
                                          └─→ Gmail 通知負責人
```

## 三、檔案分層

```
[Drive 總資料夾]/
├── 01_公文專案啟動表/          ← 第一張表 + 回應 Sheet
├── 02_專案總控表/              ← 集中管理（行政專案總控表 + 專案階段日期紀錄）
├── 03_專案資料夾/              ← 所有專案都放這
│   └── {年度}_{處室}_{專案名稱}/
│       ├── 00_原始公文與附件
│       ├── 01_計畫書與核定資料
│       ├── 02_工作分工與會議紀錄
│       ├── 03_表單與回覆資料
│       ├── 04_經費與採購核銷
│       ├── 05_活動照片與照片說明
│       ├── 06_成果資料與成果報告
│       ├── 07_公告通知與對外文字
│       ├── 08_簡報與成果展示
│       ├── 09_檢討與下次改進
│       └── 99_系統產生文件     ← Docs / Sheets 自動產出物放這
├── 04_專案階段日期新增表/      ← 第二張表 + 回應 Sheet
└── 06_Apps Script/             ← 預留給未來程式檔
```

## 四、關鍵 Apps Script 函式

| 函式 | 用途 | 觸發方式 |
|------|------|---------|
| `installFlow()` | 主安裝入口 | 手動執行 |
| `uninstallFlow()` | 清掉觸發器與 ScriptProperties | 手動執行 |
| `onIntakeSubmit(e)` | 處理公文表單送出 | Form 觸發器 |
| `onMilestoneSubmit(e)` | 處理階段日期送出 | Form 觸發器 |
| `runSelfCheck()` | 自我檢測 Drive/Sheet/Calendar | 手動執行 |
| `listAllProjects()` | 列出所有專案 | 手動執行 |
| `getProjectStatus(code)` | 查單一專案 | 手動執行 |
| `archiveProject(code)` | 標記專案為結案 | 手動執行 |

## 五、與其他系統的接點

| 對接系統 | 接點 | 方向 |
|---------|------|------|
| NotebookLM | 「NotebookLM 筆記本連結」欄位（總控表） | 手動回填 |
| Gemini | Doc 內「九、NotebookLM 建議提問」段落 | 提問模板 |
| Google Calendar | `SETTINGS.calendarId`（預設 primary） | 寫入 |
| Gmail | 自動寄信給 `承辦人Email` / `負責人Email` | 寫入 |
| Drive | 整套架構的根 | 讀寫 |

## 六、設計限制與已知問題

- **表單不收附件**：避免 Form 上傳的暫存連結與權限混亂。承辦人事後手動上傳。
- **觸發器一次只能對一張 Sheet**：兩張 Form 各自掛在自己的回應 Sheet 上。
- **ScriptProperties 是記錄 ID 的單一來源**：若手動刪除 Form 或 Sheet，要同步清掉對應的 Property（或執行 `uninstallFlow()` 再重裝）。
- **時區固定為 Asia/Taipei**：若要改其他時區，調整 `SETTINGS.timezone`。
